# Dependency Mapping Across .NET Codebases, Terraform, Dynatrace, Confluence, Log Aggregation, and the Configuration and Service Data Model (CSDM)

## Research Question

What practical tools and methodologies are being used to map dependencies across .NET codebases, Terraform configurations, Dynatrace Application Performance Management (APM) monitoring, and solution documentation in Confluence -- and how are organisations practically implementing these approaches to produce a detailed understanding of existing-state architecture and dependency trees? How do signals from log aggregation and the Configuration and Service Data Model (CSDM) extend and validate these maps? What role can locally-running Large Language Model (LLM) agents play in automating discovery and mapping across all these surfaces, given that every source will be incomplete, partially incorrect, or contain blind spots?

Supporting questions:
- What static analysis and tooling approaches exist for extracting dependency graphs from .NET codebases (project references, NuGet packages, inter-service calls, database dependencies)?
- How can Terraform configurations be parsed to produce resource and module dependency graphs, and what tools and Infrastructure as Code (IaC) analysis tools exist for this?
- How does Dynatrace APM's automated dependency discovery (Smartscape, service flow, topology) complement or conflict with statically-derived maps, and how is the data exported or queried?
- What is the role of Confluence documentation as a source of dependency information -- and what are its failure modes (staleness, incompleteness, inconsistency)?
- How do log aggregation pipelines surface runtime dependency signals that neither static analysis nor APM topology captures?
- What is CSDM in the ServiceNow context, and how does it model application, service, and infrastructure dependencies -- and where does it typically break down in practice?
- How can locally-running LLM agents automate the discovery and reconciliation of dependencies across all these surfaces?
- What strategies exist for handling the reality that all of these sources are partially wrong, incomplete, or out of date -- and how do organisations build confidence in their composite dependency maps despite this?
- What does a pragmatic "good enough" dependency map look like when starting from heterogeneous, imperfect sources?

## Scope

**In scope:**
- Static analysis tooling for .NET dependency extraction: NDepend, Depends, Roslyn-based analysers, Visual Studio architecture tools, custom scripting over `.csproj` and solution files
- IaC dependency mapping tools for Terraform: `terraform graph`, Blast Radius, Rover, and related plan-visualisation tools
- Dynatrace APM dependency discovery: Smartscape topology, service dependency graph, and topology extension patterns
- Confluence as a dependency knowledge source: Application Programming Interface (API)-driven extraction, page traversal, and staleness detection
- Log aggregation as a dependency signal source: service-to-service call patterns inferred from tracing, structured logs, or shared service identifiers in Elastic, Splunk, or Datadog
- CSDM (ServiceNow Configuration and Service Data Model): structure, common gaps, and practical use as a dependency registry
- Agent-assisted discovery: locally-running LLM agents used for code graph extraction, cross-source reconciliation, gap detection, and natural-language querying of dependency graphs
- Cross-source reconciliation strategies: how to merge dependency data from heterogeneous sources when each source is partially wrong
- Practical organisational implementation patterns: team structures, tooling stacks, governance approaches, and incremental improvement strategies
- Prior research context from this repository where relevant

**Out of scope:**
- Service-mesh tools as the primary dependency-mapping mechanism
- Security-specific dependency scanning beyond its use as a signal source
- Non-.NET languages unless the approach generalises
- Vendor feature comparisons below the level of architectural decisions
- Implementation of a specific dependency-mapping pipeline for a named organisation

**Constraints:** Publicly accessible sources only: vendor documentation, GitHub repositories, practitioner write-ups, and prior completed items in this repository. Given the operational nature of the question, practitioner accounts carry weight when they describe failure modes or governance lessons. All external sources are cited at URL level.

## Context

Large organisations accumulate architectural truth across multiple heterogeneous surfaces: source code, infrastructure declarations, runtime observability, enterprise registries, and team-maintained documentation. No single surface gives a complete picture of "what depends on what".

The practical challenge is to build a dependency map that is detailed enough for blast-radius analysis, migration sequencing, decommissioning safety, and change planning, while accepting that every source is incomplete, stale, or semantically partial.

The key constraints are:
1. **Imperfect sources** -- Terraform can diverge from the live environment; Dynatrace only shows observed and instrumented relationships; Confluence pages go stale; CSDM records lag without governance; logs and traces miss low-traffic or uninstrumented paths.
2. **Heterogeneous semantics** -- .NET project references, Terraform resource graphs, Smartscape topology, Confluence page trees, and CSDM relationship tables do not encode the same kind of dependency.
3. **Local agent capability** -- modern local agents can call deterministic extractors and APIs, build graphs, and explain conflicts, but they still depend on source quality and explicit provenance.

This research aims to identify the best current tools for each surface and synthesise a pragmatic multi-surface mapping approach that treats imperfection as a design assumption rather than a defect to be removed later.

## Approach

1. **Surface-by-surface tool audit** -- identify what each surface can produce, how it is exported, and what it misses.
2. **CSDM deep dive** -- clarify what CSDM models well and where data-quality failures appear in practice.
3. **Runtime signal analysis** -- compare tracing-based maps with log-derived evidence.
4. **Agent-assisted discovery patterns** -- identify where local agents help and where deterministic tooling should remain primary.
5. **Cross-source reconciliation strategies** -- define conflict handling, freshness tracking, and confidence scoring.
6. **Pragmatic map strategy** -- define what a "good enough" map looks like operationally.
7. **Synthesis** -- produce an evidence-backed answer, evidence map, and practical approach guide.

### Executive Summary

**[inference]** The best-supported answer is to treat dependency mapping as a graph-merge problem: combine declared edges from code and Infrastructure as Code (IaC), observed edges from runtime telemetry, and curated ownership or taxonomy records from documentation and service-management systems.

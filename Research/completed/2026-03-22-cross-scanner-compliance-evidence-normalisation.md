# Cross-Scanner Compliance Evidence and Waiver Normalisation in GitHub Actions

## Research Question

How should an organisation running multiple compliance scanners in GitHub Actions normalise evidence, severity, waiver handling, and developer-facing output so that heterogeneous tools behave like one coherent compliance system rather than a collection of unrelated failing checks?

## Scope

**In scope:**
- Evidence models for combining outputs from CodeQL, Semgrep, Checkov, Spectral, Graph Query Language (GraphQL) Inspector, Structured Query Language Fluff (SQLFluff), and similar tools that emit Static Analysis Results Interchange Format (SARIF) or comparable machine-readable results
- Waiver and suppression governance across heterogeneous scanners
- Developer-facing result surfaces in GitHub: code scanning alerts, check runs, summaries, annotations, and external evidence stores
- Mapping scanner-native severities into an organisation-level severity taxonomy
- Auditability requirements for exceptions, expiry, and policy-version traceability

**Out of scope:**
- Building a new scanner engine
- Runtime security telemetry outside GitHub Actions
- Vendor-specific dashboards that cannot be driven from GitHub Actions or open formats

**Constraints:** Public documentation, open standards where possible, and practical GitHub-native implementation paths.

## Context

The completed item `2026-03-22-compliance-scanning-gh-actions.md` concluded that broad Policy as Code (PaC) coverage in GitHub Actions is already achievable with a portfolio of specialised scanners (source: https://github.com/davidamitchell/Research/blob/main/Research/completed/2026-03-22-compliance-scanning-gh-actions.md). The unresolved problem is no longer scanner availability but scanner operating model. Without a shared evidence and waiver model, teams inherit multiple suppression syntaxes, inconsistent severities, and fragmented audit trails.

## Approach

1. Survey the output and suppression models of leading GitHub-integrated scanners.
2. Identify the common denominator formats and metadata fields needed for aggregation.
3. Evaluate GitHub-native result surfaces and their trade-offs.
4. Propose a minimum viable evidence schema and waiver lifecycle.
5. Synthesize an implementation path suitable for a central compliance platform.

### §2 Investigation

#### 2.1 Prior work and common interchange primitives

- [fact] The OASIS Static Analysis Results Interchange Format (SARIF) specification says its purpose is to reduce the cost and complexity of aggregating results from multiple analysis tools into common workflows. Source (primary): https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html
- [fact] GitHub code scanning accepts third-party SARIF 2.1.0 uploads and displays those alerts alongside native CodeQL alerts in the repository security view. Sources (primary): https://docs.github.com/en/code-security/concepts/code-scanning/about-code-scanning-alerts ; https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning
- [fact] GitHub documents that SARIF uploads need stable `ruleId` values and consistent file paths for duplicate-prevention logic to work correctly across runs. Source (primary): https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning
- [fact] GitHub uses `partialFingerprints` to match logically identical results across runs, and the `upload-sarif` action can compute `partialFingerprints` when the SARIF file omits them and the source tree is present. Sources (primary): https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning ; https://docs.github.com/en/code-security/how-tos/scan-code-for-vulnerabilities/integrate-with-existing-tools/uploading-a-sarif-file-to-github
- [fact] SARIF itself defines result `level` values such as `note`, `warning`, and `error`, and it also defines suppression objects plus fingerprint fields for result correlation. Source (primary): https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html
- [inference] SARIF is the right interchange spine for a GitHub-native compliance platform because it standardizes location, rule identity, severity presentation, suppression concepts, and result correlation, while GitHub already knows how to present that subset without custom user interface work. Basis: https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html ; https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning

#### 2.2 GitHub-native surfaces and their constraints

- [fact] GitHub says code scanning alerts can appear in the security tab and in pull requests as check results and annotations. Source (primary): https://docs.github.com/en/code-security/concepts/code-scanning/about-code-scanning-alerts
- [fact] GitHub says a pull request alert appears only when all lines identified by the alert exist in the pull request diff. Sources (primary): https://docs.github.com/en/code-security/concepts/code-scanning/about-code-scanning-alerts ; https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning
- [fact] GitHub says code scanning alerts always have a `severity` of `Error`, `Warning`, or `Note`, and CodeQL security alerts may also carry `Critical`, `High`, `Medium`, or `Low` security severity. Source (primary): https://docs.github.com/en/code-security/concepts/code-scanning/about-code-scanning-alerts
- [fact] GitHub says multiple SARIF result sets for the same commit must be differentiated by `category` or `runAutomationDetails.id`, otherwise later uploads replace earlier ones. Source (primary): https://docs.github.com/en/code-security/how-tos/scan-code-for-vulnerabilities/integrate-with-existing-tools/uploading-a-sarif-file-to-github
- [inference] GitHub code scanning is well-suited to location-aware findings that can be represented as stable SARIF results, but it is a weak fit for cross-file governance summaries, policy waivers awaiting approval, and scanner outputs that do not naturally map to source-line alerts. Basis: https://docs.github.com/en/code-security/concepts/code-scanning/about-code-scanning-alerts ; https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning

#### 2.3 Scanner-native outputs, severities, and suppression models

- [fact] Semgrep documents two ignore modes: file and folder exclusion through `.semgrepignore`, and code-level suppression through `nosemgrep` comments; Semgrep also says code-level ignores still generate a finding record that is automatically triaged as ignored. Source (primary): https://semgrep.dev/docs/ignoring-files-folders-code
- [fact] Semgrep's JSON and SARIF reference shows machine-readable results with rule identifiers, locations, messages, severities, and SARIF-compatible result blocks. Source (primary): https://semgrep.dev/docs/semgrep-appsec-platform/json-and-sarif
- [fact] Checkov's CLI reference lists machine-readable outputs including `json`, `junitxml`, `github_failed_only`, and `sarif`, and it also exposes `--create-baseline`, `--baseline`, `--skip-check`, `--skip-path`, `--soft-fail-on`, and `--hard-fail-on`. Source (primary): https://www.checkov.io/2.Basics/CLI%20Command%20Reference.html
- [fact] Checkov documents inline suppressions through `checkov:skip=<CHECK_ID>:<reason>` comments in supported configuration files. Source (primary): https://www.checkov.io/2.Basics/Suppressing%20and%20Skipping%20Policies.html
- [fact] Spectral's official CLI documentation supports formats including `json`, `junit`, `github-actions`, `sarif`, and `markdown`, and the official Spectral materials describe rule severities such as `error`, `warn`, `info`, `hint`, and `off`. Sources (primary): https://docs.stoplight.io/docs/spectral/branches/develop/9ffa04e052cc1-spectral-cli ; https://github.com/stoplightio/spectral
- [fact] GraphQL Inspector's GitHub Action annotates pull requests, exposes a total `changes` output, enables annotations by default, fails on breaking changes by default, and can permit expected breaking changes through an approval label. Source (primary): https://the-guild.dev/graphql/inspector/docs/products/action
- [fact] SQLFluff documents `-- noqa` directives, block disable and enable directives, `--disable-noqa`, `--ignore`, and `--exclude-rules` controls for lint suppression and filtering. Sources (primary): https://docs.sqlfluff.com/en/latest/configuration/ignoring_configuration.html ; https://docs.sqlfluff.com/en/stable/reference/cli.html
- [inference] The scanners share a common core of rule identity, location, message, and some notion of importance, but they diverge sharply in waiver syntax and in what their "severity" means, because GraphQL Inspector uses change impact categories, Checkov uses compliance and security thresholding, Spectral uses lint severities, and SQLFluff focuses on lint selection and suppression rather than security weighting. Basis: https://semgrep.dev/docs/ignoring-files-folders-code ; https://www.checkov.io/2.Basics/CLI%20Command%20Reference.html ; https://docs.stoplight.io/docs/spectral/branches/develop/9ffa04e052cc1-spectral-cli ; https://the-guild.dev/graphql/inspector/docs/products/action ; https://docs.sqlfluff.com/en/latest/configuration/ignoring_configuration.html
- [inference] A single flattened severity field would lose meaning across these tools, so normalization should separate user-interface display level, business severity, and merge-gate policy instead of forcing every tool into one ordinal ladder. Basis: https://docs.github.com/en/code-security/concepts/code-scanning/about-code-scanning-alerts ; https://docs.stoplight.io/docs/spectral/branches/develop/9ffa04e052cc1-spectral-cli ; https://the-guild.dev/graphql/inspector/docs/products/action ; https://www.checkov.io/2.Basics/CLI%20Command%20Reference.html

#### 2.4 Evidence normalization and waiver governance

- [fact] SARIF supports externalized suppression metadata and justification, while GitHub code scanning relies on stable rule identifiers, paths, and fingerprints to manage repeated findings across scans. Sources (primary): https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html ; https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning
- [fact] Checkov baseline files and Semgrep ignore states are scanner-specific mechanisms for suppressing known findings, not organization-wide exception records shared across tools. Sources (primary): https://www.checkov.io/2.Basics/CLI%20Command%20Reference.html ; https://semgrep.dev/docs/ignoring-files-folders-code
- [inference] The minimum viable normalized evidence record needs at least these fields: `tool`, `category`, `rule_id`, `rule_name`, `artifact_uri`, `region`, `message`, `fingerprint`, `presentation_level`, `business_severity`, `result_kind`, `workflow_category`, `policy_version`, `first_seen`, `last_seen`, and `source_urls` or `help_uri`. Basis: https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html ; https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning ; https://semgrep.dev/docs/semgrep-appsec-platform/json-and-sarif
- [inference] The minimum viable waiver record should be separate from the finding record and keyed by the normalized fingerprint plus scanner identity, with fields for `waiver_id`, `status`, `scope`, `justification`, `requested_by`, `approved_by`, `created_at`, `expires_at`, and `policy_version`, because native suppression syntaxes do not provide consistent approval or expiry semantics across tools. Basis: https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html ; https://semgrep.dev/docs/ignoring-files-folders-code ; https://www.checkov.io/2.Basics/Suppressing%20and%20Skipping%20Policies.html ; https://docs.sqlfluff.com/en/latest/configuration/ignoring_configuration.html
- [inference] Inline suppressions should be treated as local escape hatches, not the canonical system of record, because they are heterogeneous, difficult to audit centrally, and weak on expiry and approver traceability. Basis: https://semgrep.dev/docs/ignoring-files-folders-code ; https://www.checkov.io/2.Basics/Suppressing%20and%20Skipping%20Policies.html ; https://docs.sqlfluff.com/en/latest/configuration/ignoring_configuration.html
- [assumption] A central waiver registry can be implemented either as a repository-managed manifest file or as an external evidence store that the workflows query at runtime. This assumption is justified because the retrieved public documentation defines the scanner and GitHub primitives but does not mandate a single storage location for exception metadata. Justification sources: https://docs.github.com/en/code-security/concepts/code-scanning/about-code-scanning-alerts ; https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html

#### 2.5 Recommended GitHub operating model

- [inference] SARIF-capable, location-aware findings from CodeQL, Semgrep, Checkov, and Spectral should be normalized into one SARIF-aligned pipeline and uploaded to GitHub code scanning under distinct categories per tool or per policy domain, because that preserves native alert navigation while avoiding result overwrites. Basis: https://docs.github.com/en/code-security/how-tos/scan-code-for-vulnerabilities/integrate-with-existing-tools/uploading-a-sarif-file-to-github ; https://semgrep.dev/docs/semgrep-appsec-platform/json-and-sarif ; https://www.checkov.io/8.Outputs/SARIF.html ; https://docs.stoplight.io/docs/spectral/branches/develop/9ffa04e052cc1-spectral-cli
- [inference] GraphQL Inspector results belong in GitHub checks and pull request annotations rather than a forced SARIF conversion, because the official action is already built around annotations, change summaries, and `fail-on-breaking` behavior instead of SARIF upload semantics. Basis: https://the-guild.dev/graphql/inspector/docs/products/action
- [inference] SQLFluff results should initially flow into a check run or job summary unless the organization is prepared to maintain a SARIF converter, because the retrieved official SQLFluff documentation emphasizes `noqa`, rule filtering, and CLI output control rather than native SARIF emission. Basis: https://docs.sqlfluff.com/en/latest/configuration/ignoring_configuration.html ; https://docs.sqlfluff.com/en/stable/reference/cli.html
- [inference] Developer-facing output should be layered: line-level findings in code scanning or check annotations, per-job normalized tables in job summaries, and organization-level waiver and evidence history in a durable registry or dashboard. Basis: https://docs.github.com/en/code-security/concepts/code-scanning/about-code-scanning-alerts ; https://the-guild.dev/graphql/inspector/docs/products/action ; https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning
- [inference] The rollout path should start with normalization and read-only aggregation, then add central waiver enforcement, and only then make merged severity and expiry policy blocking, because category separation and fingerprint stability need to settle before strict governance depends on them. Basis: https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning ; https://github.com/davidamitchell/Research/blob/main/Research/completed/2026-03-22-compliance-scanning-gh-actions.md

### §3 Reasoning

- [inference] The strongest common denominator is not "one scanner to rule them all" but "one evidence contract across several scanners," because the sources consistently show heterogeneous tools converging on machine-readable outputs while retaining tool-specific semantics. Basis: https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html ; https://semgrep.dev/docs/semgrep-appsec-platform/json-and-sarif ; https://www.checkov.io/2.Basics/CLI%20Command%20Reference.html ; https://docs.stoplight.io/docs/spectral/branches/develop/9ffa04e052cc1-spectral-cli
- [inference] Severity should be modeled as three orthogonal concepts: presentation level for user interface placement, business severity for risk ranking, and merge-gate decision for enforcement policy. This removes the category error that occurs when one tool's `breaking` change or `warn` level is forced into another tool's security severity ladder. Basis: https://docs.github.com/en/code-security/concepts/code-scanning/about-code-scanning-alerts ; https://the-guild.dev/graphql/inspector/docs/products/action ; https://docs.stoplight.io/docs/spectral/branches/develop/9ffa04e052cc1-spectral-cli ; https://www.checkov.io/2.Basics/CLI%20Command%20Reference.html
- [inference] Waivers should be centralized even if scanners retain local suppression syntax, because auditability depends on approver identity, expiry, and policy-version traceability that inline comments and scanner-local baselines do not consistently provide. Basis: https://www.checkov.io/2.Basics/Suppressing%20and%20Skipping%20Policies.html ; https://www.checkov.io/2.Basics/CLI%20Command%20Reference.html ; https://semgrep.dev/docs/ignoring-files-folders-code ; https://docs.sqlfluff.com/en/latest/configuration/ignoring_configuration.html

### §5 Depth and Breadth Expansion

- [inference] **Technical lens:** fingerprint stability and path normalization are not cosmetic implementation details; they determine whether developers see one persistent finding or repeated alert churn. Basis: https://docs.github.com/en/code-security/reference/code-scanning/sarif-files/sarif-support-for-code-scanning
- [inference] **Governance lens:** expiry-bearing waivers matter more in a heterogeneous scanner estate because a permanent inline suppression can silently outlive the policy context that justified it. Basis: https://www.checkov.io/2.Basics/Suppressing%20and%20Skipping%20Policies.html ; https://semgrep.dev/docs/ignoring-files-folders-code ; https://docs.sqlfluff.com/en/latest/configuration/ignoring_configuration.html
- [inference] **Developer-experience lens:** forcing every finding into one blocking status check increases confusion when the underlying tools are signaling different things, whereas layered output preserves context and makes triage cheaper. Basis: https://docs.github.com/en/code-security/concepts/code-scanning/about-code-scanning-alerts ; https://the-guild.dev/graphql/inspector/docs/products/action
- [inference] **Economic lens:** central normalization reduces duplicated platform work later because each new scanner only needs an ingestion adapter plus a mapping to the shared contract, not a new waiver system and new reporting model. Basis: https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html ; https://github.com/davidamitchell/Research/blob/main/Research/completed/2026-03-22-compliance-scanning-gh-actions.md
- [inference] **Historical lens:** SARIF exists precisely because static-analysis ecosystems became fragmented; using that standard as the backbone is more defensible than inventing another proprietary result schema for the same problem. Basis: https://docs.oasis-open.org/sarif/sarif/v2.1.0/sarif-v2.1.0.html

### §6 Synthesis

- [inference] **Direct answer:** an organization should run a hub-and-adapter model in which each scanner feeds a normalized evidence contract, waivers are managed in a separate organization-owned registry, and GitHub code scanning is used only for the subset of findings that genuinely behave like stable, location-aware alerts.
